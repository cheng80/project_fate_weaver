# 0. 헌법 선언

이 문서는 구현 문서가 아니다.

이 문서는 **Project FateWeaver가 어떤 게임이어야 하는지, 어떤 게임이 되면 안 되는지, 어떤 기준으로 이후의 모든 설계와 구현 결정을 내려야 하는지**를 정의하는 프로젝트 헌법이다.

PRD, World Bible, 데이터 스키마, Flutter/Flame 아키텍처, Codex 작업 지시서는 모두 이 문서를 상위 기준으로 삼는다.

v1.2부터 이 문서는 단순 방향 선언이 아니라, **설계가 맞는지 틀렸는지 검증할 수 있는 기준**을 포함한다.

---

# 1. 프로젝트 한 줄 정의

> **Project FateWeaver는 판타지 세계의 사건, 상태, 아이템, 선택지를 온톨로지 기반 데이터로 구성하고, 플레이어가 매 런마다 자신의 선택으로 이야기를 만들어가는 Reigns식 로그라이크 이벤트 시뮬레이션이다.**

---

# 2. 핵심 목표

## 2.1 만들고자 하는 경험

플레이어가 빠른 손놀림이 아니라 **판단, 후회, 욕심, 예측, 리스크 관리**로 재미를 느끼게 한다.

플레이어가 한 판을 끝낸 뒤 이렇게 느끼는 것이 목표다.

```text
아까 성수를 아껴둘 걸.
밧줄이 있었으면 다른 선택지가 열렸을 텐데.
저주를 너무 오래 방치했다.
이번에는 평판을 높이는 루트로 가보자.
한 이벤트만 더 보고 끝내자.
```

---

## 2.2 핵심 재미

이 프로젝트의 Core Fun은 다음 네 가지다.

1. **상황 판단**
2. **대가가 있는 선택**
3. **상태 변화의 누적**
4. **매 런마다 달라지는 이야기**

---

# 3. 만들지 않을 게임

## 3.1 실시간 액션 게임이 아니다

금지 방향:

- 탄막 회피
- 빠른 반응속도
- 실시간 이동 컨트롤
- 회피 액션
- 전투 손맛 중심

이 프로젝트는 손이 빠른 플레이어보다 **판단을 잘하는 플레이어**가 유리해야 한다.

---

## 3.2 던전 맵 탐험 게임이 아니다

금지 방향:

- 대규모 타일맵 제작
- 방/문/통로 중심 던전
- 이동 탐험 자체가 핵심인 구조
- 맵 레벨 디자인 대량 제작

던전 감각은 참고할 수 있으나, 콘텐츠 단위는 맵이 아니라 **Event**이다.

---

## 3.3 전통 RPG 전투 게임이 아니다

금지 방향:

- 공격력/방어력 중심 전투
- 스킬 쿨타임
- 몬스터 HP 깎기
- 장비 강화 중심 성장
- 별도 전투 화면을 기본 루프로 편입
- 전투 카드 중심 덱빌딩화

전투는 존재할 수 있지만, MVP에서는 전투를 **사건(Event)의 한 종류**로 추상화한다.

예:

```text
오크와 전투한다 ❌
오크 습격 사건을 해결한다 ✅
```

---

## 3.4 긴 메인 스토리 게임이 아니다

금지 방향:

- 챕터형 메인 스토리
- 긴 컷신
- 고정된 서사 루트
- 후속 이벤트 체인에 과도하게 의존하는 구조

이 프로젝트는 개발자가 정해둔 이야기를 따라가는 게임이 아니라, **시스템과 선택으로 이야기가 생성되는 게임**이다.

---

# 4. 설계 원칙

## 4.1 Story Driven이 아니라 World Driven

이 프로젝트는 긴 스토리를 쓰는 것이 아니라, **세계가 작동하는 규칙**을 만든다.

```text
Story Driven:
이벤트 A → 이벤트 B → 이벤트 C

World Driven:
상태 변화 → 이벤트 확률 변화 → 선택지 변화 → 새로운 결과
```

핵심은 스토리 플래그가 아니라 **상태와 관계**다.

---

## 4.2 콘텐츠 최소 단위는 Event

콘텐츠는 맵, 몬스터, 스테이지가 아니다.

콘텐츠 최소 단위는 Event다.

예:

- 버려진 우물
- 수상한 상인
- 무너진 다리
- 저주받은 제단
- 고블린 습격
- 안개 낀 폐허
- 굶주린 마을
- 신성한 샘

---

## 4.3 카드는 능력치가 아니라 선택지다

카드 또는 아이템은 단순한 수치 보너스가 아니다.

좋지 않은 카드:

```text
공격력 +2
방어력 +3
이동속도 +20%
```

좋은 카드:

```text
밧줄 → 절벽, 우물, 함정에서 새로운 선택지 해금
성수 → 저주, 언데드, 오염된 우물 정화
횃불 → 어둠, 동굴, 유령 이벤트에서 조사 선택지 해금
지도 → 길 잃음, 우회로, 숨겨진 길 이벤트에서 사용
도둑 도구 → 잠긴 상자, 감옥, 상점 이벤트에서 사용
```

카드는 능력을 올리는 것이 아니라, **세계를 대하는 방법을 늘린다.**

---

## 4.4 선택지는 정답이 아니라 대가다

좋지 않은 선택지:

```text
A. 좋은 결과
B. 나쁜 결과
C. 의미 없음
```

좋은 선택지:

```text
A. 안전하지만 보상 적음
B. 위험하지만 큰 보상
C. 자원을 소모하지만 상태 개선
```

선택지는 항상 다음 중 하나 이상의 대가를 가져야 한다.

- 상태 감소
- 아이템 소모
- 위험 증가
- 미래 선택지 제한
- 현재 보상 포기
- 다음 이벤트 확률 변화
- 정보 부족 상태에서의 판단

---

## 4.5 상태가 스토리를 대체한다

플레이어가 선택한 결과는 긴 스토리 분기보다 **상태 변화**로 남는다.

예:

```text
성수를 사용했다 → 성수 -1
물을 마셨다 → 저주 +1
도둑을 도왔다 → 평판 -1, 돈 +3
아이를 구했다 → 평판 +2, 식량 -1
```

상태는 다음 이벤트와 선택지를 바꾼다.

---

# 5. 게임 구조

## 5.1 Core Loop

```text
Run 시작
↓
현재 상태 확인
↓
이벤트 발생
↓
선택지 3개 제시
↓
플레이어 선택
↓
결과 적용
↓
상태 변화
↓
다음 이벤트 후보 변화
↓
생존 / 실패 / 귀환 / 엔딩
```

---

## 5.2 턴 구조

이 프로젝트는 빠른 템포의 실시간 게임이 아니다.

추천 턴 구조:

```text
이벤트 카드 표시
↓
현재 상태와 보유 아이템 확인
↓
선택지 선택
↓
결과 적용
↓
다음 이벤트
```

한 턴은 짧지만, 플레이어가 생각할 시간을 가진다.

---

## 5.3 Run 구조

Run은 짧아야 한다.

MVP-1 기준 추천:

| 항목 | 기준 |
|---|---|
| 1 Run 목표 길이 | 10~20분 |
| 1 Event 소요 시간 | 10~30초 |
| 1 Run 이벤트 수 | 15~30개 |
| 초기 엔딩 | 3개 |
| 초기 지역 | 3개 |

# 6. Validation Criteria

v1.2부터 Constitution은 **검증 가능한 문서**여야 한다.

이 프로젝트가 올바른 방향으로 가고 있다고 말하려면 아래 기준을 통과해야 한다.

---

## 6.1 Core Loop 검증 기준

MVP-0 콘솔 프로토타입에서 아래 기준을 확인한다.

| 검증 항목 | 통과 기준 |
|---|---|
| 반복 플레이 의향 | 12~15개 이벤트만으로도 5회 이상 반복 테스트할 이유가 있어야 한다 |
| 선택 후 후회 | 플레이어가 최소 3회 이상 “다른 선택을 했으면 달랐겠다”고 느껴야 한다 |
| 상태 영향 | 현재 상태가 최소 30% 이상의 이벤트 선택 가치에 영향을 줘야 한다 |
| 아이템 영향 | 보유 아이템이 최소 30% 이상의 이벤트에서 선택지를 바꿔야 한다 |
| 정답 회피 | 이벤트의 절반 이상은 항상 최선의 선택지가 하나로 고정되지 않아야 한다 |
| 다음 이벤트 영향 | 선택 결과의 일부가 다음 이벤트 후보 또는 선택지에 영향을 줘야 한다 |

---

## 6.2 실패 판정 기준

아래 중 하나라도 확인되면 설계를 수정해야 한다.

- 플레이어가 대부분의 이벤트에서 정답을 바로 찾는다.
- 이벤트가 수치 손익 계산표처럼 느껴진다.
- 아이템이 단순 열쇠로만 작동하고 전략적 가치가 없다.
- 저주가 독특한 시스템이 아니라 단순 패널티 수치로만 느껴진다.
- 이벤트 반복 시 암기 게임으로 바뀐다.
- Codex가 이벤트를 많이 만들수록 품질이 떨어진다.
- 온톨로지가 재미 검증보다 문서 작업을 우선하게 만든다.
- 전투형 이벤트가 단순 `체력 -1 / 돈 +3`으로만 처리된다.

---

## 6.3 “재미 있음”의 최소 증거

다음 반응이 테스트에서 나와야 한다.

```text
이번에는 다른 아이템으로 시작해보고 싶다.
이번에는 저주를 일부러 쌓아보고 싶다.
이번에는 평판을 버리고 돈을 모아보고 싶다.
한 이벤트만 더 보고 싶다.
이 선택을 나중에 후회할 것 같다.
```

# 8. Event Quality Checklist

Codex가 생성한 이벤트는 아래 기준을 통과해야 한다.

---

## 8.1 좋은 이벤트의 필수 조건

이벤트 하나는 다음 조건을 만족해야 한다.

- 선택지 3개 이상
- 최소 2개 선택지에 명확한 대가 존재
- 최소 1개 선택지는 상태 또는 아이템 조건과 연결
- 최소 1개 결과는 미래 이벤트 확률, 상태, 아이템, 태그에 영향
- 현재 상태에 따라 선택 가치가 달라짐
- 최소 1개 선택지는 안전하지만 보상이 낮음
- 최소 1개 선택지는 위험하지만 보상이 큼
- 반복 등장 시 완전히 같은 판단만 요구하지 않음

---

## 8.2 나쁜 이벤트의 조건

아래 중 하나라도 있으면 수정 또는 폐기한다.

- 보상만 있고 대가가 없음
- 항상 최선의 선택지가 명확함
- 상태/아이템/태그와 연결되지 않음
- 결과가 즉시 수치 변화로만 끝남
- 선택지 텍스트만 다르고 구조가 동일함
- 현재 상태와 무관하게 항상 같은 선택이 유리함
- 실패해도 다음 이벤트에 영향이 없음
- 이벤트 문장이 길지만 게임적 의미가 없음

---

## 8.3 Codex 이벤트 Reject 기준

Codex가 생성한 이벤트는 아래 기준 중 하나라도 위반하면 rejected 상태로 둔다.

- 필수 필드 누락
- 존재하지 않는 태그 참조
- 선택지 3개 미만
- 모든 선택지가 같은 상태만 변경
- 대가 없는 보상 선택지 2개 이상
- 조건부 선택지 없음
- 미래 영향 없음
- 이벤트 태그와 선택지 내용 불일치
- 전투형 이벤트가 전투 감각 없이 단순 손익표로만 구성

---

# 9. Item Role Taxonomy

아이템은 단순 선택지 해금 열쇠에 머물면 안 된다.

아이템은 최소 하나 이상의 역할을 가져야 한다.

---

## 9.1 선택지 해금

특정 이벤트에서 새로운 행동을 가능하게 한다.

예:

```text
밧줄 → 절벽을 내려간다
성수 → 우물을 정화한다
도둑 도구 → 상자를 연다
```

---

## 9.2 리스크 완화

위험한 선택의 피해를 줄인다.

예:

```text
약초 → 독 피해 완화
부적 → 저주 결과 완화
방패 → 전투형 이벤트 피해 완화
```

---

## 9.3 확률 조작

결과 풀의 확률을 바꾼다.

예:

```text
지도 → 길 잃음 이벤트 실패 확률 감소
행운의 부적 → 위험 선택의 좋은 결과 확률 증가
마법서 → 마법 이벤트의 보상 확률 증가
```

---

## 9.4 정보 공개

미래 이벤트나 선택 결과 일부를 미리 보여준다.

예:

```text
지도 → 다음 지역 태그 일부 공개
점술 카드 → 다음 이벤트 위험 태그 공개
정찰 도구 → 선택지의 숨은 대가 표시
```

---

## 9.5 미래 이벤트 유도

특정 종류의 이벤트가 나오도록 가중치를 바꾼다.

예:

```text
상인의 증표 → 거래 이벤트 확률 증가
성자의 인장 → 사원/성소 이벤트 확률 증가
저주받은 유물 → 묘지/언데드 이벤트 확률 증가
```

---

## 9.6 대가 전환

손실을 다른 상태 손실로 바꾼다.

예:

```text
부적 → 저주 +1을 돈 -2로 전환
명예 훈장 → 평판 손실을 체력 손실로 전환
검은 계약서 → 체력 손실을 저주 증가로 전환
```

# 10. Curse System Policy

저주는 보조 상태가 아니라 Project FateWeaver의 핵심 변형 시스템 후보로 본다.

---

## 10.1 저주의 역할

저주는 단순 패널티가 아니다.

저주는 런의 성격을 바꾸는 축이다.

저주가 증가하면 다음이 일어난다.

- 언데드/묘지/제단 이벤트 확률 증가
- 일부 선택지 오염
- 일부 아이템 효과 반전
- 특정 보상 증가
- 위험하지만 강한 선택지 등장
- 엔딩 조건 변화

---

## 10.2 저주의 설계 방향

저주는 피해야만 하는 수치가 아니어야 한다.

좋은 저주 설계:

```text
저주가 높아지면 위험하지만 새로운 기회도 생긴다.
```

예:

```text
저주 1: 묘지 이벤트 증가
저주 2: 유령 선택지 등장
저주 3: 일부 회복 선택지 오염
저주 4: 금지된 보상 등장
저주 5: 저주 엔딩 위험
```

---

## 10.3 검증 질문

MVP-0에서 확인해야 한다.

- 저주를 낮추는 것이 항상 정답인가?
- 일부 상황에서는 저주를 감수할 이유가 있는가?
- 저주가 다음 이벤트를 눈에 띄게 바꾸는가?
- 저주가 플레이어의 런 전략을 바꾸는가?

# 11. Combat Policy

## 11.1 전투의 위치

전투는 판타지 세계에서 자연스럽게 등장할 수 있다.

하지만 MVP 단계에서 전투는 독립 시스템이 아니다.

전투는 다음처럼 정의한다.

```text
전투 시스템 ❌
전투형 이벤트 ✅
```

즉, `고블린 습격`, `늑대 무리`, `도적 매복`, `언데드 출현` 같은 사건은 별도 전투 화면으로 진입하지 않고, 선택지와 결과를 통해 처리한다.

---

## 11.2 MVP 전투 원칙

MVP에서 전투는 아래 규칙을 따른다.

- 별도 전투 화면을 만들지 않는다.
- 턴제 전투 미니게임을 만들지 않는다.
- 공격/방어 수치 싸움을 핵심 루프로 넣지 않는다.
- 전투 카드를 별도 덱으로 분리하지 않는다.
- 전투는 Event-Choice-State 구조 안에서 처리한다.
- 아이템, 상태, 평판, 저주가 전투형 이벤트 선택지를 바꿀 수 있다.

---

## 11.3 좋은 전투형 이벤트 조건

전투형 이벤트는 다음을 포함해야 한다.

- 위협이 명확함
- 회피/협상/대응/희생 중 최소 2개 이상 선택 가능
- 단순 `체력 -1 / 돈 +3`으로 끝나지 않음
- 상처, 피로, 공포, 평판, 저주 등 후유증 가능
- 아이템 또는 상태가 선택지를 바꿈
- 전투를 회피할 수 있는 선택지도 존재 가능
- 전투 결과가 다음 이벤트에 영향을 줄 수 있음

---

## 11.4 전투형 이벤트 예시

```yaml
id: goblin_ambush
name: 고블린 습격
event_tags:
  - combat
  - goblin
  - ambush
danger_tags:
  - physical
  - group_enemy

choices:
  - id: flee
    text: 도망친다
    result:
      food: -1
      event_weight:
        bandit: +1

  - id: negotiate
    text: 은화로 달랜다
    requires_status:
      money: 2
    result:
      money: -2
      reputation: +1

  - id: fight_with_dagger
    text: 단검으로 맞선다
    requires_item: dagger
    result:
      health: -1
      money: +3
      fatigue: +1

  - id: use_torch
    text: 횃불로 위협한다
    requires_item: torch
    result:
      torch: -1
      reputation: +1
      event_weight:
        beast: -1
```

- Flutter 앱은 검증된 데이터를 안정적으로 로드한다.

# 13. 이벤트 설계

## 13.1 단일 이벤트

단일 이벤트는 하나의 주요 상황으로 구성된다.

예:

```text
버려진 우물
```

선택지:

- 지나간다
- 물을 마신다
- 성수로 정화한다
- 밧줄로 내려간다

---

## 13.2 합성 이벤트

합성 이벤트는 여러 의미 블록의 조합으로 만들어진다.

예:

```text
장소 + 환경 + 존재
```

조합 예:

```text
숲 + 보름달 + 늑대
시장 + 축제 + 도둑
폐허 + 안개 + 유령
우물 + 저주 + 성수
```

합성 이벤트는 런타임 완전 자동 생성보다 **개발 단계의 콘텐츠 생성 보조**로 사용한다.

즉:

```text
온톨로지 조합
↓
Codex 이벤트 초안 생성
↓
개발자 검토
↓
YAML 저장
```

# 14. 데이터 관리 헌법

## 14.1 YAML이 Single Source of Truth

모든 핵심 게임 데이터는 YAML로 관리한다.

```text
data/
├── regions.yaml
├── statuses.yaml
├── tags.yaml
├── items.yaml
├── events.yaml
├── choices.yaml
└── endings.yaml
```

Flutter 앱은 YAML을 직접 수정하지 않는다.

개발 파이프라인:

```text
YAML 작성
↓
Python 검증
↓
JSON Export
↓
Flutter assets 로드
```

---

## 14.2 코드에 콘텐츠를 하드코딩하지 않는다

금지:

```dart
if (eventId == 'abandoned_well') {
  ...
}
```

권장:

```text
event data → rule engine → result
```

콘텐츠는 데이터로 관리하고, 코드는 룰만 관리한다.

# 20. 판단 기준

앞으로 모든 아이디어는 아래 기준을 통과해야 한다.

## 20.1 추가해도 되는 기능

- 선택의 의미를 강화한다.
- 상태 변화와 연결된다.
- 이벤트 제작을 쉽게 만든다.
- 온톨로지 구조와 맞는다.
- Flutter + Flame 하이브리드와 충돌하지 않는다.
- Codex와 Python 자동화로 관리 가능하다.
- 전투가 Event-Choice-State 루프 안에서 처리된다.
- MVP-0 검증을 방해하지 않는다.

---

## 20.2 추가하면 안 되는 기능

- 실시간 조작을 요구한다.
- 맵 제작 비용을 크게 늘린다.
- 콘텐츠를 코드에 묶는다.
- LLM이 게임 판정을 하게 만든다.
- 상태 시스템과 무관하다.
- 선택을 단순 정답 찾기로 만든다.
- 전투가 별도 게임처럼 커진다.
- 전투가 핵심 루프를 대체한다.
- 재미 검증 전에 문서/온톨로지 제작량만 늘린다.

# 21. 최종 헌법 조항

## 제1조

이 프로젝트는 반사신경 게임이 아니라 판단 게임이다.

## 제2조

콘텐츠의 최소 단위는 Event이다.

## 제3조

카드와 아이템은 능력치가 아니라 선택지를 여는 도구다.

## 제4조

스토리는 직접 쓰는 것이 아니라 상태와 선택으로 생성된다.

## 제5조

게임 판정은 항상 룰 기반이다.

## 제6조

AI는 개발과 콘텐츠 제작을 돕지만, 런타임 판정을 대신하지 않는다.

## 제7조

YAML은 게임 콘텐츠의 Single Source of Truth다.

## 제8조

Python은 콘텐츠 검증과 시뮬레이션의 핵심 도구다.

## 제9조

Flutter는 UI, Flame은 연출, Dart 룰 엔진은 판정을 담당한다.

## 제10조

PRD와 구현은 이 Constitution을 위반할 수 없다.

## 제11조

전투는 존재할 수 있으나, MVP에서는 전투형 이벤트로만 추상화한다.

## 제12조

별도 전투 시스템은 Constitution 개정 없이는 도입하지 않는다.

## 제13조

이 프로젝트의 설계는 문서량이 아니라, 최소 이벤트 세트로 Core Loop가 재미있다는 것을 증명해야 한다.

## 제14조

MVP 이전 단계에서는 12~15개 이벤트 콘솔 프로토타입으로 먼저 루프를 검증한다.

## 제15조

Codex가 생성한 콘텐츠는 Event Quality Checklist를 통과하지 못하면 게임 데이터에 포함하지 않는다.

## 제16조

저주는 단순 패널티가 아니라 런을 변형하는 핵심 시스템 후보로 검증한다.
