# 2. Constitution의 역할

Project FateWeaver Constitution은 다음 역할을 해야 한다.

## 2.1 프로젝트 정체성 정의

-   이 게임이 무엇인지 정의한다.
-   이 게임이 무엇이 아닌지 정의한다.
-   핵심 재미를 고정한다.
-   프로젝트가 방향을 잃었을 때 되돌아올 기준점이 된다.

## 2.2 의사결정 기준

새로운 기능이나 아이디어가 나왔을 때 판단 기준이 된다.

검토 질문:

``` text
이 기능은 Constitution을 강화하는가?
아니면 Constitution을 흐리는가?
```

## 2.3 PRD의 상위 문서

PRD는 이 Constitution을 기반으로 작성되어야 한다.

PRD는 세부 요구사항을 정의하지만, Constitution은 그 요구사항이 따라야 할
철학과 경계를 정의한다.

## 2.4 World Bible의 상위 문서

World Bible은 세계관을 확장한다.

하지만 World Bible은 Constitution의 아래 원칙을 위반하면 안 된다.

-   Story Driven이 아니라 World Driven
-   Event 중심
-   State 중심
-   Rule Based
-   Data Driven

## 2.5 데이터 스키마의 기준

YAML 데이터 구조, 태그 체계, 이벤트 구조, 아이템 구조는 Constitution에서
정의한 철학을 따라야 한다.

## 2.6 Codex 작업 지시의 기준

Codex에게 전달되는 구현 요청은 Constitution을 기준으로 검증되어야 한다.

Codex가 제안한 구조가 Constitution을 위반하면 구현하지 않는다.

## 2.7 콘텐츠 제작 기준

이벤트, 아이템, 상태, 지역, 태그를 만들 때 Constitution이 제작 가이드
역할을 한다.

## 2.8 설계 변경 통제 기준

이후 새로운 아이디어가 나왔을 때 Constitution과 충돌한다면 다음 중
하나를 선택해야 한다.

1.  아이디어를 폐기한다.
2.  아이디어를 Constitution에 맞게 수정한다.
3.  Constitution 자체를 명시적으로 개정한다.

암묵적으로 방향을 바꾸지 않는다.

# 3. 이번 Office Hour에서 원하는 피드백

이번 리뷰는 칭찬이나 아이디어 확장이 목적이 아니다.

원하는 것은 **비판적 검증**이다.

특히 아래 관점으로 검토해 달라.

------------------------------------------------------------------------

## 3.1 이 Constitution은 충분히 구체적인가?

-   PRD로 확장하기 충분한가?
-   모호한 선언이 많은가?
-   실제 결정 기준으로 사용할 수 있는가?
-   개발 중 충돌 상황에서 판단 기준으로 쓸 수 있는가?

------------------------------------------------------------------------

## 3.2 이 Constitution은 너무 과한가?

-   MVP 전에 지나치게 복잡한가?
-   온톨로지 설계가 과한가?
-   AI First 개발 철학이 실제 개발을 느리게 만들 가능성은 없는가?
-   문서 중심으로 과하게 흐를 위험은 없는가?

------------------------------------------------------------------------

## 3.3 핵심 재미가 명확한가?

현재 핵심 재미는 다음으로 정의되어 있다.

-   상황 판단
-   대가가 있는 선택
-   상태 변화의 누적
-   매 런마다 달라지는 이야기

이 정의가 충분한가?

부족하다면 무엇이 빠졌는가?

------------------------------------------------------------------------

## 3.4 Event 중심 구조가 타당한가?

콘텐츠 최소 단위를 Event로 정의했다.

검토 질문:

-   Event가 정말 이 게임의 최소 콘텐츠 단위가 맞는가?
-   Event만으로 반복 플레이를 만들 수 있는가?
-   Event 작성 비용이 과소평가되어 있지는 않은가?
-   합성 이벤트는 실제로 생산성을 높이는가, 아니면 복잡도만 높이는가?

------------------------------------------------------------------------

## 3.5 카드/아이템 철학이 타당한가?

현재 카드와 아이템은 능력치 버프가 아니라 선택지를 여는 도구로 정의되어
있다.

검토 질문:

-   이 구조가 장기적으로 재미를 만들 수 있는가?
-   선택지를 여는 카드만으로 수집 욕구가 생기는가?
-   능력치형 카드가 일부 필요하지는 않은가?
-   카드와 아이템을 같은 개념으로 둘지 분리해야 하는가?

------------------------------------------------------------------------

## 3.6 State 시스템이 충분한가?

MVP 상태는 다음 5개다.

-   체력
-   식량
-   돈
-   평판
-   저주

검토 질문:

-   5개가 적절한가?
-   너무 단순한가?
-   너무 관리 부담이 큰가?
-   이 상태들로 충분히 다양한 이벤트 변화를 만들 수 있는가?
-   `저주`는 핵심 축으로 충분히 매력적인가?

------------------------------------------------------------------------

## 3.7 Ontology 중심 설계가 적절한가?

현재 프로젝트는 World Ontology를 핵심 설계로 둔다.

검토 질문:

-   이 방식이 실제 콘텐츠 제작에 도움이 되는가?
-   너무 데이터 설계 중심으로 흐를 위험은 없는가?
-   더 단순한 구조가 가능한가?
-   RDF/OWL 같은 복잡한 지식공학으로 가지 않고 YAML 기반 관계 설계로
    충분한가?

------------------------------------------------------------------------

## 3.8 AI First 개발 철학이 현실적인가?

현재 방향은 Codex를 적극 활용하는 AI First Development다.

검토 질문:

-   Codex가 어떤 단계에서 가장 큰 가치를 줄 수 있는가?
-   AI가 오히려 문서와 데이터의 일관성을 해칠 위험은 없는가?
-   AI 활용 범위를 더 명확히 제한해야 하는가?
-   AI가 만든 이벤트 초안을 검증할 기준이 충분한가?

------------------------------------------------------------------------

## 3.9 MVP 범위가 적절한가?

현재 MVP 범위:

  항목               수량
  -------------- --------
  Region                3
  Status                5
  Item                 10
  Event            40\~50
  Combat Event       5\~8
  Ending                3
  Tag              20\~30

검토 질문:

-   이 범위가 MVP로 적절한가?
-   이벤트 40\~50개가 과한가?
-   전투형 이벤트 5\~8개가 적절한가?
-   더 작은 검증 단위가 가능한가?
-   Flutter UI 구현 전에 콘솔 시뮬레이터로 먼저 검증해야 하는가?

------------------------------------------------------------------------

## 3.10 Combat Policy가 타당한가?

v1.1 Constitution에서는 전투를 다음과 같이 정의했다.

``` text
MVP: 전투 없음 또는 전투형 이벤트로 추상화
v1 이후: 전투형 이벤트 확장
고급 버전: 별도 미니 전투 룰 검토
```

검토 질문:

-   전투를 MVP에서 별도 시스템으로 구현하지 않는 결정이 타당한가?
-   전투형 이벤트만으로 판타지 감각을 충분히 줄 수 있는가?
-   전투를 추상화하면 재미가 너무 약해지지 않는가?
-   별도 전투 시스템을 도입해야 하는 시점은 언제인가?
-   전투가 Core Fun을 흐릴 위험은 없는가?
-   `CombatEventResolver`가 필요한가, 일반 `ChoiceResolver`로 충분한가?
