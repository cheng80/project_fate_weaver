# 0. 헌법 선언

이 문서는 구현 문서가 아니다.

이 문서는 **Project FateWeaver가 어떤 게임이어야 하는지, 어떤 게임이
되면 안 되는지, 어떤 기준으로 이후의 모든 설계와 구현 결정을 내려야
하는지**를 정의하는 프로젝트 헌법이다.

PRD, World Bible, 데이터 스키마, Flutter/Flame 아키텍처, Codex 작업
지시서는 모두 이 문서를 상위 기준으로 삼는다.

# 1. 프로젝트 한 줄 정의

> **Project FateWeaver는 판타지 세계의 사건, 상태, 아이템, 선택지를
> 온톨로지 기반 데이터로 구성하고, 플레이어가 매 런마다 자신의 선택으로
> 이야기를 만들어가는 Reigns식 로그라이크 이벤트 시뮬레이션이다.**

# 2. 핵심 목표

## 2.1 만들고자 하는 경험

플레이어가 빠른 손놀림이 아니라 **판단, 후회, 욕심, 예측, 리스크
관리**로 재미를 느끼게 한다.

플레이어가 한 판을 끝낸 뒤 이렇게 느끼는 것이 목표다.

``` text
아까 성수를 아껴둘 걸.
밧줄이 있었으면 다른 선택지가 열렸을 텐데.
저주를 너무 오래 방치했다.
이번에는 평판을 높이는 루트로 가보자.
한 이벤트만 더 보고 끝내자.
```

------------------------------------------------------------------------

## 2.2 핵심 재미

이 프로젝트의 Core Fun은 다음 네 가지다.

1.  **상황 판단**
2.  **대가가 있는 선택**
3.  **상태 변화의 누적**
4.  **매 런마다 달라지는 이야기**

------------------------------------------------------------------------

# 3. 만들지 않을 게임

이 문서는 프로젝트의 경계를 분명히 하기 위해, 하지 않을 것을 먼저
정의한다.

## 3.1 실시간 액션 게임이 아니다

금지 방향:

-   탄막 회피
-   빠른 반응속도
-   실시간 이동 컨트롤
-   회피 액션
-   전투 손맛 중심

이 프로젝트는 손이 빠른 플레이어보다 **판단을 잘하는 플레이어**가
유리해야 한다.

------------------------------------------------------------------------

## 3.2 던전 맵 탐험 게임이 아니다

금지 방향:

-   대규모 타일맵 제작
-   방/문/통로 중심 던전
-   이동 탐험 자체가 핵심인 구조
-   맵 레벨 디자인 대량 제작

던전 감각은 참고할 수 있으나, 콘텐츠 단위는 맵이 아니라 **Event**이다.

------------------------------------------------------------------------

## 3.3 전통 RPG 전투 게임이 아니다

금지 방향:

-   공격력/방어력 중심 전투
-   스킬 쿨타임
-   몬스터 HP 깎기
-   장비 강화 중심 성장

전투는 존재할 수 있지만, 전투는 **사건(Event)의 한 종류**로 추상화한다.

예:

``` text
오크와 전투한다 ❌
오크 습격 사건을 해결한다 ✅
```

------------------------------------------------------------------------

## 3.4 긴 메인 스토리 게임이 아니다

금지 방향:

-   챕터형 메인 스토리
-   긴 컷신
-   고정된 서사 루트
-   후속 이벤트 체인에 과도하게 의존하는 구조

이 프로젝트는 개발자가 정해둔 이야기를 따라가는 게임이 아니라,
**시스템과 선택으로 이야기가 생성되는 게임**이다.

------------------------------------------------------------------------

# 4. 설계 원칙

## 4.1 Story Driven이 아니라 World Driven

이 프로젝트는 긴 스토리를 쓰는 것이 아니라, **세계가 작동하는 규칙**을
만든다.

``` text
Story Driven:
이벤트 A → 이벤트 B → 이벤트 C

World Driven:
상태 변화 → 이벤트 확률 변화 → 선택지 변화 → 새로운 결과
```

핵심은 스토리 플래그가 아니라 **상태와 관계**다.

------------------------------------------------------------------------

## 4.2 콘텐츠 최소 단위는 Event

콘텐츠는 맵, 몬스터, 스테이지가 아니다.

콘텐츠 최소 단위는 Event다.

예:

-   버려진 우물
-   수상한 상인
-   무너진 다리
-   저주받은 제단
-   고블린 습격
-   안개 낀 폐허
-   굶주린 마을
-   신성한 샘

------------------------------------------------------------------------

## 4.3 카드는 능력치가 아니라 선택지다

카드 또는 아이템은 단순한 수치 보너스가 아니다.

좋지 않은 카드:

``` text
공격력 +2
방어력 +3
이동속도 +20%
```

좋은 카드:

``` text
밧줄 → 절벽, 우물, 함정에서 새로운 선택지 해금
성수 → 저주, 언데드, 오염된 우물 정화
횃불 → 어둠, 동굴, 유령 이벤트에서 조사 선택지 해금
지도 → 길 잃음, 우회로, 숨겨진 길 이벤트에서 사용
도둑 도구 → 잠긴 상자, 감옥, 상점 이벤트에서 사용
```

카드는 능력을 올리는 것이 아니라, **세계를 대하는 방법을 늘린다.**

------------------------------------------------------------------------

## 4.4 선택지는 정답이 아니라 대가다

좋지 않은 선택지:

``` text
A. 좋은 결과
B. 나쁜 결과
C. 의미 없음
```

좋은 선택지:

``` text
A. 안전하지만 보상 적음
B. 위험하지만 큰 보상
C. 자원을 소모하지만 상태 개선
```

선택지는 항상 다음 중 하나 이상의 대가를 가져야 한다.

-   상태 감소
-   아이템 소모
-   위험 증가
-   미래 선택지 제한
-   현재 보상 포기
-   다음 이벤트 확률 변화

------------------------------------------------------------------------

## 4.5 상태가 스토리를 대체한다

플레이어가 선택한 결과는 긴 스토리 분기보다 **상태 변화**로 남는다.

예:

``` text
성수를 사용했다 → 성수 -1
물을 마셨다 → 저주 +1
도둑을 도왔다 → 평판 -1, 돈 +3
아이를 구했다 → 평판 +2, 식량 -1
```

상태는 다음 이벤트와 선택지를 바꾼다.

------------------------------------------------------------------------

# 5. 게임 구조

## 5.1 Core Loop

``` text
Run 시작
↓
현재 상태 확인
↓
이벤트 발생
↓
선택지 3개 제시
↓
플레이어 선택
↓
결과 적용
↓
상태 변화
↓
다음 이벤트 후보 변화
↓
생존 / 실패 / 귀환 / 엔딩
```

------------------------------------------------------------------------

## 5.2 턴 구조

이 프로젝트는 빠른 템포의 실시간 게임이 아니다.

추천 턴 구조:

``` text
이벤트 카드 표시
↓
현재 상태와 보유 아이템 확인
↓
선택지 선택
↓
결과 적용
↓
다음 이벤트
```

한 턴은 짧지만, 플레이어가 생각할 시간을 가진다.

------------------------------------------------------------------------

## 5.3 Run 구조

Run은 짧아야 한다.

MVP 기준 추천:

  항목                기준
  ------------------- ----------
  1 Run 목표 길이     10\~20분
  1 Event 소요 시간   10\~30초
  1 Run 이벤트 수     15\~30개
  초기 엔딩           3개
  초기 지역           3개

# 6. World Ontology

## 6.1 온톨로지의 목적

이 프로젝트에서 온톨로지는 학술적 RDF/OWL 구현이 아니다.

게임 세계를 관리하기 위한 **관계형 설계도**다.

목적:

-   이벤트, 아이템, 상태, 태그의 관계를 명확히 한다.
-   이벤트 추가 시 일관성을 유지한다.
-   AI(Codex)가 콘텐츠를 생성할 때 기준을 제공한다.
-   Python 검증 도구가 데이터 누락과 불균형을 찾을 수 있게 한다.
-   Flutter 앱은 검증된 데이터를 안정적으로 로드한다.

------------------------------------------------------------------------

## 6.2 핵심 개체

``` text
World
├── Region
├── Event
├── Choice
├── Result
├── Item

└── Rule
```

-   평판 요구 선택

## 6.6 Result

Result는 반드시 룰 엔진이 계산한다.

LLM이 결과를 결정하지 않는다.

# 7. 이벤트 설계

## 7.1 단일 이벤트

단일 이벤트는 하나의 주요 상황으로 구성된다.

예:

``` text
버려진 우물
```

선택지:

-   지나간다
-   물을 마신다
-   성수로 정화한다
-   밧줄로 내려간다

------------------------------------------------------------------------

## 7.2 합성 이벤트

합성 이벤트는 여러 의미 블록의 조합으로 만들어진다.

예:

``` text
장소 + 환경 + 존재
```

조합 예:

``` text
숲 + 보름달 + 늑대
시장 + 축제 + 도둑
폐허 + 안개 + 유령
우물 + 저주 + 성수
```

합성 이벤트는 런타임 완전 자동 생성보다 **개발 단계의 콘텐츠 생성
보조**로 사용한다.

즉:

``` text
온톨로지 조합
↓
Codex 이벤트 초안 생성
↓
개발자 검토
↓
YAML 저장
```

------------------------------------------------------------------------

## 7.3 이벤트 제작 원칙

좋은 이벤트는 다음 조건을 가진다.

-   선택지가 명확하다.
-   정답이 하나가 아니다.
-   현재 상태에 따라 의미가 달라진다.
-   아이템이 있으면 새로운 선택지가 열린다.
-   결과가 다음 이벤트에 영향을 준다.
-   한 번 보고 끝나는 스토리보다 여러 상태와 조합될 수 있다.

------------------------------------------------------------------------

# 8. 데이터 관리 헌법

## 8.1 YAML이 Single Source of Truth

모든 핵심 게임 데이터는 YAML로 관리한다.

``` text
data/
├── regions.yaml
├── statuses.yaml
├── tags.yaml
├── items.yaml
├── events.yaml
├── choices.yaml
└── endings.yaml
```

Flutter 앱은 YAML을 직접 수정하지 않는다.

개발 파이프라인:

``` text
YAML 작성
↓
Python 검증
↓
JSON Export
↓
Flutter assets 로드
```

------------------------------------------------------------------------

## 8.2 코드에 콘텐츠를 하드코딩하지 않는다

금지:

``` dart
if (eventId == 'abandoned_well') {
  ...
}
```

권장:

``` text
event data → rule engine → result
```

콘텐츠는 데이터로 관리하고, 코드는 룰만 관리한다.

------------------------------------------------------------------------

# 9. AI First Development

## 9.1 Codex의 역할

Codex는 단순 코드 생성기가 아니다.

Codex는 다음 역할을 한다.

-   공동 게임 디자이너
-   설계 리뷰어
-   이벤트 초안 작성자
-   데이터 스키마 보조자
-   Flutter 구현 보조자
-   Python 도구 작성자
-   테스트 코드 작성자
-   리팩터링 파트너

------------------------------------------------------------------------

## 9.2 AI가 해도 되는 일

-   이벤트 초안 생성
-   선택지 초안 생성
-   결과 후보 제안
-   문장 윤문
-   태그 추천
-   데이터 누락 탐지
-   코드 생성
-   테스트 생성
-   문서 정리

------------------------------------------------------------------------

## 9.3 AI가 하면 안 되는 일

-   런타임에서 게임 결과 결정
-   플레이어 보상 임의 생성
-   확률을 즉석에서 변경
-   저장 데이터 수정
-   밸런스 값 무단 변경
-   룰 엔진 판정 대체

게임 판정은 항상 룰 기반이어야 한다.

------------------------------------------------------------------------

## 9.4 온디바이스 LLM 정책

앱 내 소형 LLM은 MVP에서 사용하지 않는다.

미래 옵션으로만 고려한다.

허용 역할:

-   이벤트 문장 윤문
-   반복 문장 변주
-   런 종료 모험담 요약
-   도감 설명 보강

금지 역할:

-   결과 판정
-   선택지 생성
-   보상 생성
-   상태 변경

------------------------------------------------------------------------

# 10. Python Toolchain

Python은 콘텐츠 제작과 검증의 핵심 도구다.

## 10.1 validate_data.py

검사 항목:

-   존재하지 않는 태그 참조
-   존재하지 않는 아이템 참조
-   이벤트 선택지 부족
-   사용처 없는 아이템
-   등장하지 않는 이벤트
-   너무 자주 쓰이는 태그
-   결과가 없는 선택지
-   소모품인데 소모 처리 없는 선택지

------------------------------------------------------------------------

## 10.2 generate_mermaid.py

자동 생성:

-   아이템 → 이벤트 관계도

# 11. Flutter + Flame 헌법

## 11.1 Flutter의 역할

Flutter는 게임의 앱 구조와 UI를 담당한다.

-   홈 화면
-   이벤트 카드 UI
-   선택지 UI
-   상태바
-   결과 화면
-   도감
-   설정
-   저장/로드
-   광고/상점은 추후

------------------------------------------------------------------------

## 11.2 Flame의 역할

Flame은 실시간 게임 엔진으로 과도하게 사용하지 않는다.

역할:

-   카드 등장 애니메이션
-   이벤트 전환 연출
-   파티클
-   상태 변화 이펙트
-   배경 분위기 연출
-   간단한 캐릭터 실루엣 연출

핵심 판정은 Flame이 아니라 Dart 룰 엔진이 담당한다.

------------------------------------------------------------------------

## 11.3 Dart 룰 엔진

핵심 클래스 후보:

-   EventEngine
-   ChoiceResolver
-   StateManager
-   InventoryManager
-   TagMatcher
-   RunSimulator
-   ResultApplier
-   EventSelector

룰 엔진은 UI/Flame과 분리되어야 한다.

# 15. 판단 기준

앞으로 모든 아이디어는 아래 기준을 통과해야 한다.

## 15.1 추가해도 되는 기능

-   선택의 의미를 강화한다.
-   상태 변화와 연결된다.
-   이벤트 제작을 쉽게 만든다.
-   온톨로지 구조와 맞는다.
-   Flutter + Flame 하이브리드와 충돌하지 않는다.
-   Codex와 Python 자동화로 관리 가능하다.

------------------------------------------------------------------------

## 15.2 추가하면 안 되는 기능

-   실시간 조작을 요구한다.
-   맵 제작 비용을 크게 늘린다.
-   콘텐츠를 코드에 묶는다.
-   LLM이 게임 판정을 하게 만든다.
-   상태 시스템과 무관하다.
-   선택을 단순 정답 찾기로 만든다.

------------------------------------------------------------------------

# 16. 최종 헌법 조항

## 제1조

이 프로젝트는 반사신경 게임이 아니라 판단 게임이다.

## 제2조

콘텐츠의 최소 단위는 Event이다.

## 제3조

카드와 아이템은 능력치가 아니라 선택지를 여는 도구다.

## 제4조

스토리는 직접 쓰는 것이 아니라 상태와 선택으로 생성된다.

## 제5조

게임 판정은 항상 룰 기반이다.

## 제6조

AI는 개발과 콘텐츠 제작을 돕지만, 런타임 판정을 대신하지 않는다.

## 제7조

YAML은 게임 콘텐츠의 Single Source of Truth다.

## 제8조

Python은 콘텐츠 검증과 시뮬레이션의 핵심 도구다.

## 제9조

Flutter는 UI, Flame은 연출, Dart 룰 엔진은 판정을 담당한다.

## 제10조

PRD와 구현은 이 Constitution을 위반할 수 없다.
